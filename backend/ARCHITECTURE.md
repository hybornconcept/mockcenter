# ExamNow — Backend Architecture Plan

> Nigeria's #1 CBT Exam Prep Platform  
> Backend: Cloudflare Workers · Hono.js · Neon PostgreSQL · Drizzle ORM · Hyperdrive · Better Auth

---

## Stack Overview

| Layer | Technology | Purpose |
|---|---|---|
| Runtime | Cloudflare Workers | Edge compute — fast, global, serverless |
| Framework | Hono.js | Lightweight web framework built for Workers |
| Database | Neon (PostgreSQL) | Serverless Postgres — scales to zero |
| ORM | Drizzle ORM | Type-safe SQL, fully edge-compatible |
| Connection pool | Hyperdrive | Cloudflare DB proxy — caches Neon connections |
| Auth | Better Auth | Session-based auth, OAuth, roles, CSRF protection |
| Validation | Zod + Valibot | Schema validation at all API boundaries |
| Payments | Paystack | Nigerian payment gateway (NGN) |
| Environment | .env + Wrangler | Dev and production env separation |

## Session Persistence — Resume Feature (Core Design)

This is the most critical feature. Here is the full architecture for how a user can close the app, log out, come back days later, and resume exactly where they stopped.

### How it works — step by step

```
1. SESSION CREATION  (POST /api/practice/sessions)
   ─────────────────────────────────────────────
   → Backend selects questions for this session
   → Saves question_ids[] in order (fixed, never reshuffled after creation)
   → Saves timer_remaining_secs = total_questions × time_per_question
   → Sets current_question_index = 0
   → Sets answer_snapshot = {}
   → Sets status = 'active'
   → Returns full session object + first question to frontend

2. DURING THE SESSION  (PATCH /api/practice/sessions/:id/answer)
   ─────────────────────────────────────────────
   → Every time user answers a question:
     - Backend updates answer_snapshot[questionId] = { selectedOptionId, timeSpent, flagged }
     - Updates current_question_index
     - Updates active_time_secs += timeSpent
   → Every 30 seconds (timed mode):
     - Frontend sends heartbeat: PATCH /api/practice/sessions/:id/heartbeat
     - Body: { timer_remaining_secs: <current_value> }
     - Backend saves timer_remaining_secs to DB
   → This means even if user's browser crashes, we lose at most 30 seconds of timer

3. USER LEAVES / PAUSES  (PATCH /api/practice/sessions/:id/pause)
   ─────────────────────────────────────────────
   → Body: { timer_remaining_secs, current_question_index, answer_snapshot }
   → Backend sets:
     - status = 'paused'
     - paused_at = now()
     - all resume fields saved
   → User can now close browser, log out, switch device — data is safe

4. USER COMES BACK  (GET /api/practice/sessions/active)
   ─────────────────────────────────────────────
   → Returns all sessions with status = 'active' | 'paused' for current user
   → Frontend checks on every app load — if active session exists, shows resume banner

5. USER RESUMES  (GET /api/practice/sessions/:id/resume)
   ─────────────────────────────────────────────
   → Returns:
     {
       session: { id, mode, status, timer_remaining_secs, current_question_index, active_time_secs },
       questions: [ ...ordered array of question objects with options ],
       answer_snapshot: { questionId: { selectedOptionId, timeSpent, flagged } },
       current_question: { ...full question at current_question_index }
     }
   → Frontend rebuilds exact state:
     - Timer starts from timer_remaining_secs
     - Navigator grid shows answered/flagged/unanswered from answer_snapshot
     - Current question shown at current_question_index
     - All previous answers pre-filled
   → Backend sets status = 'active', clears paused_at

6. SESSION COMPLETION  (POST /api/practice/sessions/:id/submit)
   ─────────────────────────────────────────────
   → Backend reads answer_snapshot
   → Writes all answers to practice_answers table (permanent record)
   → Calculates results + runs AI engine
   → Sets status = 'completed', completed_at = now()
   → Returns full results + AI verdicts

7. SESSION TIMEOUT (timed mode only)
   ─────────────────────────────────────────────
   → If user resumes a timed session and timer_remaining_secs <= 0:
     - Backend auto-submits with current answer_snapshot
     - Sets status = 'timed_out'
   → If user is mid-session and loses internet:
     - Frontend stores answer_snapshot in localStorage as backup
     - On reconnect, syncs with backend via PATCH /api/practice/sessions/:id/sync
```

### Resume State Fields on `practice_sessions` table explained

| Field | Type | What it stores |
|---|---|---|
| `question_ids` | `uuid[]` | Fixed ordered list of questions — never changes after creation |
| `answer_snapshot` | `jsonb` | `{ "q-uuid": { "option": "opt-uuid", "secs": 38, "flagged": false } }` |
| `current_question_index` | `integer` | Index into `question_ids[]` — which question they're on |
| `timer_remaining_secs` | `integer` | Countdown timer value — saved every 30s and on pause |
| `active_time_secs` | `integer` | Running total of time actually spent answering |
| `paused_at` | `timestamp` | When the session was paused/user left |
| `current_question_started_at` | `timestamp` | When current question was opened — to calculate time_spent accurately on resume |
| `status` | `text` | `active` \| `paused` \| `completed` \| `abandoned` \| `timed_out` |

### Frontend responsibilities for resume

```
On every page load:
  → GET /api/practice/sessions/active
  → If session found with status = active | paused:
    → Show resume banner: "You have an unfinished Biology session — Resume?"

On resume click:
  → GET /api/practice/sessions/:id/resume
  → Rebuild all state from response:
    - Set timer to timer_remaining_secs
    - Populate navigator grid from answer_snapshot
    - Navigate to current_question_index
    - Pre-fill any answered options
  → Start timer from where it left off

On browser close / navigate away (beforeunload):
  → PATCH /api/practice/sessions/:id/pause (fire-and-forget)
  → Also save answer_snapshot to localStorage as offline backup
```

## Auth Flow (Better Auth)

```
1. SIGN UP
   POST /api/auth/sign-up/email { name, email, password, referral_code? }
   → Better Auth creates user + hashed password in DB
   → Trigger (afterSignUp hook):
       - Generate unique referral_code → save to users table
       - If referral_code provided: create referrals row (status=signed_up)
       - Give new user 20 welcome credits (if referred)
       - Create notification_settings row with defaults
       - Send welcome notification

2. SIGN IN
   POST /api/auth/sign-in/email { email, password }
   → Better Auth validates, creates session row in DB
   → Returns httpOnly session cookie (secure: true in production)

3. PROTECTED ROUTES
   → auth.middleware.ts reads cookie → calls Better Auth session validation
   → Injects { user, session } into Hono context (c.var.user)
   → Returns 401 if session missing or expired

4. SIGN OUT
   POST /api/auth/sign-out
   → Better Auth deletes session from DB
   → Clears cookie
   → NOTE: Any active practice session is auto-paused before sign-out
```

## Credit Flow

```
1. User selects package       GET  /api/credits/packages
2. Initiate payment           POST /api/credits/initiate { packageId }
   → Create pending transaction in DB
   → Call Paystack: initialize transaction
   → Return { authorization_url, reference }
3. User pays on Paystack page
4. Paystack fires webhook     POST /api/credits/webhook
   → Verify HMAC-SHA512 signature (reject if invalid)
   → Find pending transaction by reference
   → credit user.credit_balance atomically (DB transaction)
   → Update transaction.status = 'success'
   → If user has a pending referral: fire referral reward
   → Send notification: "120 credits added to your account!"
5. Manual verify fallback     POST /api/credits/verify { reference }
   → For cases where webhook was missed
   → Calls Paystack verify API directly
```

## Referral Reward Flow

```
1. User A's referral link:  examenow.ng/ref/EXAMCHUKZ
2. User B visits link       → Frontend stores code in localStorage
3. User B signs up          POST /api/auth/sign-up/email { ..., referral_code: "EXAMCHUKZ" }
   → Referrals row created: { referrer: A, referred: B, status: signed_up }
   → User B gets +20 welcome credits
   → Notification to A: "Emeka just signed up using your link! Earn 50 credits when they buy."
4. User B buys credits      → Paystack webhook fires
   → Find referral where referred_id = B AND status = signed_up
   → Update status → purchased → rewarded (single DB transaction)
   → Add 50 credits to User A atomically
   → Notification to A: "🎁 You earned 50 credits! Chisom made their first purchase."
5. Tier upgrades:
   → After every reward, check referrer's total successful referrals
   → Starter (0–4): 50 credits/referral
   → Hustler (5–14): 75 credits/referral
   → Champion (15+): 100 credits/referral + cash withdrawal option
```

## AI Analytics Engine (`src/lib/ai.ts`)

A deterministic computation engine — no LLM calls, runs at the edge on every session completion.

### Inputs
- All `practice_answers` for the session (time_spent_secs, is_correct, question_id)
- Historical sessions for this user (last 10)
- Subject benchmarks (target speed per exam type)

### Outputs
```typescript
type AIVerdict = {
  avg_speed_per_question: number        // seconds
  speed_verdict: 'too_fast' | 'ideal' | 'slightly_slow' | 'too_slow'
  speed_message: string                 // e.g. "You averaged 22s — rushing. Slow down."
  accuracy_rate: number                 // 0–100
  accuracy_trend: 'improving' | 'stable' | 'declining'
  accuracy_delta: number                // change vs last 5 sessions
  weakest_topics: string[]              // sorted by accuracy ascending
  strongest_topics: string[]            // sorted by accuracy descending
  predicted_score: number               // projected out of total_questions
  readiness_score: number               // 0–100 composite
  readiness_delta: number               // change vs last readiness score
  recommendations: Recommendation[]    // array of action items
}

type Recommendation = {
  priority: 1 | 2 | 3
  subject: string
  message: string
  action: string  // e.g. "Practice Organic Chemistry without timer"
}
```

### Readiness Score Formula
```
readiness = (
  weighted_subject_accuracy  × 0.50 +
  improvement_trend          × 0.20 +
  speed_score                × 0.15 +
  consistency_score          × 0.15
) × 100

where:
  weighted_subject_accuracy = average accuracy across all practiced subjects
  improvement_trend         = slope of last 10 session scores
  speed_score               = 1 - abs(avg_speed - target_speed) / target_speed (clamped 0–1)
  consistency_score         = 1 - (std_dev_of_scores / 100)
```

## Validation Strategy

### Zod — used for:
- `src/env.ts` — validates all env vars at startup, throws if any missing
- Auth endpoints — sign-up, sign-in, reset password schemas
- Credit/payment — package IDs, Paystack webhook body
- Query params — pagination, filters, sort order

### Valibot — used for:
- `practice.validator.ts` — session create, answer submit, pause, heartbeat
- `bookmarks.validator.ts` — add, update, collection mutations
- `referral.validator.ts` — referral actions

### Why both?
Zod has the best TypeScript DX and ecosystem compatibility (Better Auth uses it internally). Valibot compiles to a significantly smaller bundle and parses faster — critical for the practice answer endpoint which fires on every single question answer during a session.

## Security

| Concern | Solution |
|---|---|
| Auth | Better Auth — httpOnly cookies, CSRF tokens, session expiry |
| Secrets | Cloudflare Secrets manager — never in code or wrangler.toml |
| Paystack webhook | HMAC-SHA512 signature verification before any DB write |
| SQL injection | Impossible — Drizzle uses parameterised queries only |
| Race conditions on credits | All credit mutations wrapped in DB transactions |
| Referral fraud | UNIQUE(referrer_id, referred_id) prevents duplicate rewards |
| Rate limiting | 60 req/min on auth, 300 req/min on practice, 1000 req/min others |
| CORS | Only APP_URL origins allowed — no wildcard in production |
| Session hijacking | httpOnly + Secure + SameSite=Strict cookies in production |
